## 五个核心用户流程

### 1. 建议与行动

- 入口：本地开发版首页的“发生了什么”或“决定下一步”。
- 步骤：
  1. 选择一条最重要变化或建议。
  2. 打开详情并核对来源、时间和证据。
  3. 只保留一个下一步、预期产物和停止条件。
- 结果：得到一条可复核、可停止的行动，而不是无限待办清单。
- 当前状态：现有 v1.2 能力保留；v1.2.1 首页与建议发现性收敛属于 S11，尚未完成。
- 边界：证据不足时只进入待复核，不直接提升为事实或应用。

### 2. 资产与主题

- 入口：“层级资产”与“哪些主题在变化”。
- 步骤：
  1. 先选择项目、来源或时间范围。
  2. 比较主题增强、减弱以及资产之间的上下层关系。
  3. 打开代表记录，确认关系不是由空筛选或单条异常造成。
- 结果：知道哪些资产值得继续积累，哪些主题需要补证据或降噪。
- 当前状态：现有浏览能力保留；v1.2.1 可编辑资产与主题工作台属于 S12，尚未完成。
- 边界：聚类和关系是分析线索，不自动成为长期记忆结论。

### 3. 编辑与提案

- 入口：建议详情、提案工作区，或本地 atlasctl proposals/apply 命令。
- 步骤：
  1. 确认目标文件、允许的字段和证据。
  2. 检查差异、验证命令、风险和回滚方案。
  3. 由人类授权后应用；验证失败立即回滚或进入待修订。
- 结果：把候选修改变成有授权、有验证、有恢复路径的变更。
- 当前状态：本地 proposal apply/rollback 边界已存在；v1.2.1 编辑工作台和完整闭环属于 S12-S13，尚未完成。
- 边界：托管网站只读；raw、凭证和未授权目标永远不能作为 apply target。

### 4. 复盘与迭代

- 入口：复盘、总结与迭代视图，并结合时间轴和搜索。
- 步骤：
  1. 选择复盘周期并查看主题、机会、低价值循环和决策变化。
  2. 回到证据核对增强、衰退或冲突的原因。
  3. 形成下一步；只有需要修改长期状态时才生成提案。
- 结果：得到有证据的周期总结、下一步和可控迭代候选。
- 当前状态：现有复盘能力保留；v1.2.1 搜索到行动闭环和模型收敛属于 S13-S14，尚未完成。
- 边界：活跃度、词频和经济 proxy 只是辅助信号，不是个人诊断或收益承诺。

### 5. 同步与备份

- 入口：owner-daily dry-run，以及 chatgpt、codex、future-agent 的 sync dry-run。
- 步骤：
  1. 先查看计划、真实输入、写入范围和停止条件。
  2. 按来源同步，再生成派生分析和 Atlas 快照。
  3. 通过审计后才进入备份；最终统一交付前不推送远端。
- 结果：真实来源、派生分析、运行证据和恢复材料保持可追溯。
- 当前状态：现有 dry-run 路径保留；v1.2.1 多来源同步、Owner 管线和最终恢复验收属于 S06-S10、S16，尚未完成。
- 边界：没有真实输入就停止；不生成伪数据，不提交凭证，不覆盖或删除 raw。

## 交付规则

- 每个 Task 都要有最小实现、验证结果、风险、回滚和本地 commit；一次 run 最多完成一个 Task。
- 每个 Stage 完成后复审；全部 Task 完成后再整体复审并修复问题。
- 只有最终验收通过，才统一 push GitHub main、重装本地应用并部署线上版本；中间不创建 branch 或 PR。
- 原始 TaskPack、Roadmap、恢复 bundle、不可变 manifest 和必要运行资料保留到整体任务完成。
