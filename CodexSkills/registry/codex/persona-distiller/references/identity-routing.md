# Identity routing

The seven build-time entries are research and registration topology, not seven personality stereotypes or a runtime user menu. Each maps to evidence topology, capability boundaries, strategy, work products and task adapters.

- Single identity: one family at 100%.
- Multi identity: two or more main families, normalized to 1.0.
- At runtime, the person Skill automatically selects or combines its distilled facets from the current task. Never ask the caller to select an identity, number, or weight.
- Weight conflicts are resolved by domain relevance and hard boundaries, not arithmetic averaging. Example: a technical falsification threshold may dominate a designer's exploratory breadth for a safety-critical engineering decision.
- `subject_origin` is independent: public/private/self/fictional/historical. Private and self require authority; fictional requires canon/version provenance; historical requires source-tier rules.

Runtime readiness states: `ready`, `provisional`, `unavailable`. The internal router must not activate an unavailable facet; it must stay within evidenced facets or offer an evidence-expansion build.
