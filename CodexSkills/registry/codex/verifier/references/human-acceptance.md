# 真人式使用验收

“像真人”不是让 agent 随机点击，而是用有目标的角色、现实节奏、错误和恢复行为完成任务，并用确定性结果确认成功。

## Persona 最小字段

- 角色/权限、经验水平、设备/输入方式、语言/时区。
- 要完成的工作与成功结果。
- 可容忍时间、失败成本、常见误操作。
- 前置知识：首次用户不能被假定知道隐藏入口或内部术语。

## 每条关键旅程的双通道

1. **确定性通道**：稳定 locator、明确断言、trace/HAR/数据校验，可重复运行。
2. **探索通道**：视觉理解、文案歧义、路径发现、纠错体验、布局和反馈延迟。

探索通道发现缺陷，确定性通道负责复现和 release gate。LLM 说“看起来成功”不能代替结果断言。

## 必测真人行为

- 首次进入：是否知道下一步、空状态是否提供行动、错误是否能理解。
- 中途犹豫：停留、返回/前进、刷新、切换 tab、离开再回来。
- 输入失误：空值、格式错、粘贴、前后空格、Unicode/emoji、超长、重复点击、双提交。
- 状态变化：会话过期、权限变更、数据被他人修改、网络慢/断后恢复。
- 多设备：至少目标主视口与一个小屏；触控目标、滚动、软键盘遮挡、方向变化按适用性。
- 键盘：Tab/Shift+Tab 顺序、可见焦点、Enter/Space、Esc、无 keyboard trap，关键功能不用鼠标也可完成。
- 可访问反馈：标签、错误关联、状态提示、对比度、缩放、动态内容；自动 axe 只是起点，必须激活 modal/menu/错误状态后再测。

## 记录指标

| 指标 | 定义 |
|---|---|
| task_success | 完成 / 部分完成 / 失败，必须有结果证据 |
| time_on_task | 从看到任务到结果确认；注明冷/热状态 |
| error_count | 用户可感知错误与恢复次数 |
| assistance_count | 需要文档、提示或隐藏知识的次数 |
| dead_end_count | 无明确下一步或无法返回的状态 |
| feedback_latency | 操作到可理解反馈的时间 |
| accessibility_blockers | 键盘、焦点、标签、缩放等阻断数 |

这些指标用于版本比较和问题定位，不应凭空设绝对“行业标准”。验收阈值来自产品目标或基线。

## 视觉证据

- 关键状态截图 + trace/video；视觉复刻需有 reference 与容差规则。
- 像素 diff 仅是信号，动态内容、字体渲染和抗锯齿必须设置 mask/容差并由语义断言兜底。
- gallery 每项写测试 ID、预期、实际、状态、严重度和证据哈希。

## 无障碍结论边界

自动规则不能证明完整 WCAG 合规。报告应区分：`automated-checks passed`、`manual keyboard checks passed`、`screen-reader/user testing not run`。未做人类/辅助技术测试时不得写“完全无障碍”。
